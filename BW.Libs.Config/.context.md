# BW.Libs.Config - Agent Context

## Project Overview
This is a **BruceWare library NuGet package** created from the `bw-lib` template. It follows BruceWare's standards for internal library development and distribution via AWS CodeArtifact.

## Architecture

### Package Distribution
- **Private NuGet feed**: AWS CodeArtifact (`bruceware-libs`)
- **Package ID**: `BW.Libs.Config`
- **Target Framework**: .NET 10.0
- **Repository**: https://github.com/timuribragimovkz/BW.Libs.Config

### Key Components
- **Abstractions/**: Public interfaces and contracts
- **Contracts/**: Value objects and DTOs
- **Components/**: Core implementations (internal)
- **DiExtensions/**: Dependency injection registration extensions
- **Factory/**: Factory patterns for service creation (if applicable)
- **Exceptions/**: Custom exception types

## Development Workflow

### 1. Version Management (Versionize)
- Uses **semantic versioning** via conventional commits
- See `VERSIONIZE.md` for detailed workflow
- Run `versionize --dry-run` before releasing

**Commit Format**:
```bash
feat: add new feature       # Minor bump (1.0.0 → 1.1.0)
fix: resolve bug           # Patch bump (1.0.0 → 1.0.1)
BREAKING CHANGE: ...       # Major bump (1.0.0 → 2.0.0)
docs: update README        # No bump
```

### 2. Publishing Workflow
- See `PUBLISHING.md` for full details
- Uses `publish.ps1` or `publish.sh` scripts
- Automatically reads CodeArtifact URL from `nuget.config`

**Quick Publish**:
```bash
./publish.ps1              # PowerShell
./publish.sh               # Bash
```

### 3. Configuration Files

#### `nuget.config`
Defines private NuGet sources:
- `bruceware-libs`: AWS CodeArtifact for BW packages
- `nuget.org`: Public packages fallback

#### `.versionize`
Configures which `.csproj` files to version and changelog format.

#### `Directory.Build.props`
Shared MSBuild properties for all BW.Libs projects:
- Nullable reference types enabled
- XML documentation generation
- Source Link for debugging

#### `.editorconfig`
Code style rules (C# conventions, indentation, etc.)

## Design Principles

### 1. Clean API Surface
- Public interfaces in `Abstractions/`
- Internal implementations in `Components/`
- Use `internal sealed` for implementation classes

### 2. Dependency Injection First
- All services registered via `IServiceCollection` extensions
- Place in `DiExtensions/` folder
- Example:
```csharp
public static IServiceCollection AddMyLib(this IServiceCollection services)
{
    services.AddSingleton<IMyService, MyService>();
    return services;
}
```

### 3. Domain Isolation
- Use generic type parameters for domain separation (e.g., `IBWCache<TDomain>`)
- Prevents key collisions across different consumers

### 4. Validation & Error Handling
- Create specific exception types in `Exceptions/`
- Throw meaningful exceptions with clear messages
- Validate inputs early

## Building & Testing

### Build
```bash

```

Generates `.nupkg` in `bin/Release/` due to `GeneratePackageOnBuild=true`.

### Restore
```bash
dotnet restore
```

Pulls dependencies from CodeArtifact and NuGet.org.

## Common Tasks

### Add New Dependency
```bash
dotnet add package PackageName --version X.Y.Z
```

### Update Version Manually
Edit `.csproj`:
```xml
<Version>1.2.3</Version>
```

Or use Versionize:
```bash
versionize --release-as 1.2.3
```

### Check CodeArtifact Connection
```bash
aws codeartifact get-authorization-token \
  --domain bruceware \
  --domain-owner 560719246675 \
  --region eu-central-1 \
  --query authorizationToken \
  --output text
```

## Important Notes

- **Never commit AWS tokens** - They expire after 12 hours
- **Use conventional commits** - Otherwise Versionize won't bump versions
- **Test locally before publishing** - Create a test project that references your package
- **Follow .NET naming conventions** - PascalCase for public APIs
- **Document public APIs** - XML docs are generated automatically

## Testing Philosophy

### Integration Tests by Default
- **Default approach**: Write integration tests with real infrastructure (AWS DynamoDB, Redis, etc.)
- **Unit tests with mocks**: Only when explicitly requested
- **Test project structure**:
  - `Framework/AwsCredentials/`: AWS credentials parsing from `~/.aws/credentials`
  - `Framework/DiExtensions/`: DI setup helpers for test infrastructure
  - `TestVessels/`: Test configuration vessels
  - `TestServices/`: Services that consume config engine for testing
- **Real AWS resources**: Tests use actual DynamoDB tables, credentials from file
- **Test project template**: BW NUnit Test Project (see template standards)

## Agent Instructions

When modifying this project:
1. Read `VERSIONIZE.md` and `PUBLISHING.md` for workflow context
2. Maintain existing code style (see `.editorconfig`)
3. Add XML documentation to all public members
4. Use conventional commits for all changes
5. Do not manually edit `CHANGELOG.md` - let Versionize handle it
6. Run `dotnet build` to verify changes before committing
7. When adding features, update `README.md` with usage examples
8. **Write integration tests by default** - only mock when explicitly asked
